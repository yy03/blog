# 3. lambda表达式

/**
 * 函数式接口：
 * 加上@FunctionalInterface注解的接口为函数式接口，在该接口中有且只能有一个普通方法，但是可以有多个默认实现方法和多个静态实现方法
 * 如果接口中只有一个普通方法，则java底层默认也会加上该注解，表明是函数式接口
 */
@FunctionalInterface
interface Foo {
    public int add(int x, int y);

    default int add1(int x, int y) {
        System.out.println("add1......");
        return x+y;
    }

    default int add2(int x, int y) {
        System.out.println("add2");
        return x+y;
    }

    public static int add3(int x, int y){
        System.out.println("add3");
        return x + y;
    }

    public static int add4(int x, int y) {
        System.out.println("add4");
        return x + y;
    }
}
/**
 * Lambda表达式的使用规则：只对函数式接口有效，即该接口中有且只有一个普通方法，那么这样就不必关心接口中的方法名，只需要参数和方法体即可。
 */
public class LambdaExpression {

    public static void main(String[] args) {
        Foo foo = (x, y) -> {
            System.out.println("add");
            return x+y;
        };

        System.out.println(foo.add(1, 2));
        System.out.println(foo.add1(2, 3));
        System.out.println(foo.add2(3, 4));

        // 静态方法属于类
        System.out.println(Foo.add3(4, 5));
        System.out.println(Foo.add4(5, 6));
    }
}

# 4. Lock接口

Lock接口是java.util.concurrent.locks包下的一个接口，可以实现线程的同步，相比于synchronized关键字，synchronized是隐式锁，可以自动上锁和释放锁；而Lock是手动锁，需要手动上锁和释放锁，使用Lock而不使用synchronized的好处是，Lock可以减小锁的粒度，会更加高效。Lock接口下常用的实现类为ReentrantLock，下面使用ReentrantLock和LambdaExpression对买票代码进行改写。

import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;
/**
 * 资源类
 */
class Ticket {

    private int number = 30;
    private Lock lock = new ReentrantLock();
    //操作
    public void sale() {
        lock.lock();
        try {
            if (number > 0) {
                System.out.println(Thread.currentThread().getName() + "卖出第" + number-- + "张票，还剩" + number + "张票");
            }
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            lock.unlock();
        }
    }
}

# 4.3 使用Lock改写ThreadNotifyDemo

再上边例子中，线程之间的通信使用的是synchronized + wait + notifyAll，那么使用Lock怎么实现线程之间的通信呢？

在JDK帮助文档中可以找到描述：

  • Lock implementations provide more extensive locking operations than can be obtained using synchronized methods and statements. They allow more flexible structuring, may have quite different properties, and may support multiple associated Conditionobjects.
  • Condition factors out the Object monitor methods (wait, notifyand notifyAllinto distinct objects to give the effect of having multiple wait-sets per object, by combining them with the use of arbitrary Lockimplementations. Where a Lock replaces the use of synchronized methods and statements, a Condition replaces the use of the Object monitor methods.

意思就是使用Lock + condition.await + condition.signalAll.替换掉原来使用的synchronized + wait + notifyAll

代码改写：

class Resource {
    private int numebr = 0;
    private Lock lock = new ReentrantLock();
    private Condition condition = lock.newCondition();
    public void increment() {
        lock.lock();
        try {
            while (numebr != 0) {
                condition.await();
            }
            numebr++;
            System.out.println(Thread.currentThread().getName() + "\t" + numebr);
            condition.signalAll();
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            lock.unlock();
        }

    }

    public void decrement() {
        lock.lock();
        try {
            while (numebr == 0 ) {
                condition.await();
            }
            numebr--;
            System.out.println(Thread.currentThread().getName() + "\t" + numebr);
            condition.signalAll();
        } catch (InterruptedException e) {
            e.printStackTrace();
        } finally {
            lock.unlock();
        }
    }
}

# 5. 线程间定制化调用通信

在改写生产者消费者代码时，为什么要使用Lock + await + signalAll呢？使用原来的也可以实现功能，那么新的写法有什么优点呢？

需要了解的是，notify是与synchronized搭配使用的，而使用notify会唤醒一个随机的线程，如果要实现线程之间的精确通知顺序访问，使用synchronized + wait + notify是无法实现的，这时就需要使用到Lock + Condition

package com.study.juc.advanced;

/**
 * 问题描述：
 *          有三个线程ABC，线程A打印5条数据后通知线程B，
 *                        线程B打印10条数据后通知线程C，
 *                        线程C打印15条数据后通知线程A；
 *                        连续10次。
 *
 *  注意点：要做到线程之间的精确通知，要注意标志位的修改
 */

import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

/**
 * 资源类
 */

class Resource1 {
    private int flag = 1; //1 A, 2 B,3 C
    private Lock lock = new ReentrantLock();
    Condition condition1 = lock.newCondition(); // 对应线程A
    Condition condition2 = lock.newCondition(); // 对应线程B
    Condition condition3 = lock.newCondition(); // 对应线程C

    public void print() {
        lock.lock();
        try {
            String name = Thread.currentThread().getName();
            if ("A".equals(name)) {
                while (flag != 1) {
                    condition1.await();
                }
                for (int i = 0; i < 5; i++) {
                    System.out.println(Thread.currentThread().getName() + ",i = " + i);
                }
                flag = 2;
                condition2.signal();
            } else if ("B".equals(name)) {
                while (flag != 2) {
                    condition2.await();
                }
                for (int i = 0; i < 10; i++) {
                    System.out.println(Thread.currentThread().getName() + ",i = " + i);
                }
                flag = 3;
                condition3.signal();
            } else {
                while (flag != 3) {
                    condition3.await();
                }
                for (int i = 0; i < 15; i++) {
                    System.out.println(Thread.currentThread().getName() + ",i = " + i);
                }
                flag = 1;
                condition1.signal();
            }
        } catch (InterruptedException e) {
            e.printStackTrace();
        } finally {
            lock.unlock();
        }

    }
}

    public static void main(String[] args) {
        Resource1 resource1 = new Resource1();

        }, "B").start();

  • 标准访问，请问先打印邮件还是短信？ 先邮件后短信
  • 邮件方法暂停4秒钟，请问先打印邮件还是短信？ 先邮件后短信
  • 新增一个普通方法hello()，请问先打印邮件还是hello？ 先hello后邮件
  • 两部手机，请问先打印邮件还是短信？ 先短信后邮件
  • 两个静态同步方法，1部手机，请问先打印邮件还是短信？ 先邮件后短信
  • 两个静态同步方法，2部手机，请问先打印邮件还是短信？ 先邮件后短信
  • 1个普通同步方法，1个静态同步方法，1部手机，请问先打印邮件还是短信？ 先短信后邮件
  • 1个普通同步方法，1个静态同步方法，2部手机，请问先打印邮件还是短信？ 先短信后邮件

解释：

​ 对于普通方法，不需要获得锁即可执行；而对于普通同步方法，是对当前对象加锁，因此两部手机时，是两个不同的对象，两个不同的锁，所以会先发送 短信后发送邮件；对于静态同步方法，是对当前类加锁，所以无论有几部手机，只要访问的是静态同步方法获得的都是同一把锁，所以都会先发送邮件后发送短信；而类锁和对象锁是两把不同的锁。

# 6. 不安全集合解决方案

# 6.1 List

ArrayList是不安全的；解决方案：

  • 使用Vector，缺点是效率低，不经常使用
  • Collections.synchronizedList(ArrayList arraylist)
  • 使用CopyOnWriteArrayList，juc包下的安全集合，写的效率低，读的效率高

CopyOnWrite：写时复制，其核心思想是，如果有多个调用者（Callers）同时访问相同的资源（如内存或者是磁盘上的数据存储），他们会共同获取相同的指针指向相同的资源，直到某个调用者修改资源内容时，系统才会真正复制一份专用副本（private copy）给该调用者，而其他调用者所见到的最初的资源仍然保持不变。这过程对其他的调用者都是透明的（transparently）。此做法主要的优点是如果调用者没有修改资源，就不会有副本（private copy）被创建，因此多个调用者只是读取操作时可以共享同一份资源。

通俗易懂的讲，写入时复制技术就是不同进程在访问同一资源的时候，只有更新操作，才会去复制一份新的数据并更新替换，否则都是访问同一个资源。CopyOnWriteArrayList是如何工作的呢？简单来说，就是平时查询的时候，都不需要加锁，随便访问，只有在更新的时候，才会从原来的数据复制一个副本来，然后修改这个副本，最后把原数据替换成当前的副本。修改操作的同时，读操作不会被阻塞，而是继续读取旧的数据。这点要跟读写锁区分一下。读写锁是遵循写与写互斥、读写互斥、读读不互斥的原则，而copyOnWrite则是写写互斥、读写不互斥的原则。

CopyOnWrite容器即写时复制容器，往一个容器中添加元素的时候，不直接往当前容器object[] 添加，而是先将当前容器object[] 进行Copy，复制出来一个新的容器object[] newElements里添加元素，添加完元素之后，再将原容器的引用指向新的容器。这样的好处是可以并发的读不需要加锁，因为当前容器里不会添加元素。所以CopyOnWrite容器也是一种读写分离的思想，读和写不同的容器。

查看CopyOnWriteArrayList的add()方法和get()方法源码：

public boolean add(E e) {
	final ReentrantLock lock = this.lock;
	lock.lock();
	try {
		Object[] elements = getArray();
		int len = elements.length;
		Object[] newElements = Arrays.copyOf(elements, len + 1);
		newElements[len] = e;
		setArray(newElements);
		return true;
	} finally {
		lock.unlock();
	}
    
public E get(int index) {
	return get(getArray(), index);
}

可以看到在使用get方法时，是不需要获得锁的，而add方法是需要获得锁才能进行写时复制。这样就做到了写时复制，读写分离。

# 7. Callable接口

Callable和Runnable的区别：

  • Callable抛出异常，Runnable不抛出异常
  • Callable有返回值
  • 实现Callable要重写call方法，实现Runnable要重写run方法
lass MyThread implements Callable<Integer> {

    @Override
    public Integer call() throws Exception {
        System.out.println(Thread.currentThread().getName() + "---------come in call");
        Thread.sleep(4000);
        return 1024;
    }
}

public class CallableTest {
    public static void main(String[] args) throws ExecutionException, InterruptedException {
        FutureTask futureTask = new FutureTask(new MyThread());
        new Thread(futureTask, "A").start();
        System.out.println(futureTask.get());
        System.out.println("主线程------------");

    }

Thread的有参构造方法中只能传入Runnable类型，那么实现Callable怎么开启线程？

需要使用FutureTask，在该类的有参构造中传入实现Callable接口的类，而FutureTask又实现了Runnable接口，使用到了对象适配器的设计模式。

Callable使用的细节：

FutureTask调用get方法获得返回值时，会阻塞主线程，所以一般get方法都放在最后。

                    System.out.println(Thread.currentThread().getName() + "\t抢到了车位");
                    Thread.sleep(1000);
                    System.out.println(Thread.currentThread().getName() + "\t离开了车位");
                } catch (InterruptedException e) {
                    e.printStackTrace();
                } finally {
                    semaphore.release();
                }
            }, String.valueOf(i)).start();
        }
    }
}

class Resource {

    private ReadWriteLock readWriteLock = new ReentrantReadWriteLock();

    public void read() {

        try {
            readWriteLock.readLock().lock();
            System.out.println(Thread.currentThread().getName() + "\t读取资源");
            Thread.sleep(500);
            System.out.println(Thread.currentThread().getName() + "\t读取完成");
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            readWriteLock.readLock().unlock();
        }

    }

    public void write() {

        try {
            readWriteLock.writeLock().lock();
            System.out.println(Thread.currentThread().getName() + "\t写入资源");
            Thread.sleep(500);
            System.out.println(Thread.currentThread().getName() + "\t写入完成");
        } catch (Exception e) {
            e.printStackTrace();
        } finally {
            readWriteLock.writeLock().unlock();
        }
    }

}

public class ReadWriteLockDemo {
    public static void main(String[] args) {
        Resource resource = new Resource();

        for (int i = 1; i <= 5; i++) {
            new Thread(() -> {
                resource.read();
            },String.valueOf(i)).start();
        }

        for (int i = 1; i <= 5; i++) {
            new Thread(() -> {
                resource.write();
            },String.valueOf(i)).start();
        }
    }
}

# 10. 线程池

# 10.1 线程池的创建

线程池的接口为Executor，其子接口为ExecutorService，一般使用ExecutorService。使用Executors来创建线程池。常用方法：

  • Executors.newFixedThreadPool：创建一个固定数量大小的线程池，当线程任务数超过固定大小时，未执行的线程任务进行阻塞队列，等待线程池调度。
    • 缺点：该方式创建线程池时，会使用LinkedBlockingQueue无界队列默认构造方法，该默认构造方法大小为Integer.MAX_VALUE，所以当线程任务超多时，可能会堆积大量的请求，从而导致 OOM。但一般的系统直接使用该方式即可。
    • 解决方式：直接用new ThreadPoolExecutor的方式创建线程池，指定具体的参数信息。
  • Executors.newSingleThreadExecutor：创建一个单例线程池, 只存在一个线程运行, 多的线程任务进行阻塞状态。缺点同上
  • Executors.newCachedThreadPool：创建一个缓存线程池，当线程数过大，可回收多余线程，如线程数不足，创建新线程。缺点同上

# 10.2 原理分析

所有的线程创建的底层都是调用ThreadPoolExecutor的构造方法来创建线程池，聊聊ThreadPoolExecutor构造方法中的7个重要参数

    public ThreadPoolExecutor(int corePoolSize,
                              int maximumPoolSize,
                              long keepAliveTime,
                              TimeUnit unit,
                              BlockingQueue<Runnable> workQueue,
                              ThreadFactory threadFactory,
                              RejectedExecutionHandler handler)

  • corePoolSize：线程池中的常驻核心线程数

  • maximumPoolSize：线程池中能够容纳同时执行的最大线程数，此值必须大于等于1

  • keepAliveTime：多余的空闲线程的存活时间，当前池中线程数量超过corePoolSize时，当空闲时间达到keepAliveTime时，多余线程会被销毁直到只剩下corePoolSize个线程为止

  • unit：keepAliveTime的单位

  • workQueue：任务队列，被提交但尚未执行的任务，通过线程池的 execute() 方法提交的 Runnable 对象将存储在该参数中。其采用阻塞队列实现。

    • 注意有界队列和无界队列的区别：如果使用有界队列，当队列饱和时并超过最大线程数时就会执行拒绝策略；而如果使用无界队列，因为任务队列永远都可以添加任务，所以设置 maximumPoolSize 没有任何意义。

  • threadFactory：表示生成线程池中工作线程的线程工厂，用于创建线程，一般默认

  • handler：拒绝策略，表示当队列满了，并且工作线程大于等于线程池的最大线程数（maximumPoolSize）时如何来拒绝请求执行的runnable的策略

    • 当线程池的线程数达到最大线程数时，需要执行拒绝策略。拒绝策略需要实现 RejectedExecutionHandler 接口，并实现 rejectedExecution(Runnable r, ThreadPoolExecutor executor) 方法。不过 Executors 框架已经为我们实现了 4 种拒绝策略：

      AbortPolicy（默认）：丢弃任务并抛出 RejectedExecutionException 异常。 CallerRunsPolicy：由调用线程处理该任务。 DiscardPolicy：丢弃任务，但是不抛出异常。可以配合这种模式进行自定义的处理方式。 DiscardOldestPolicy：丢弃队列最早的未处理任务，然后重新尝试执行任务。
